## Главный вопрос: как дать практику, не превращая доклад в туториал?

Доклад — это ИСТОРИЯ. Практика должна вырастать из истории естественно, а не как вставной урок.

### Решение: "инструменты в контексте истории + шпаргалка на вынос"

**В докладе:** каждый инструмент появляется в момент, когда он нужен по сюжету:
- dmesg → когда спускаемся в ОС (Р3)
- гипотезы → когда ищем причину (Р4)
- воспроизведение → когда нужно доказать (Р6)
- git bisect → когда ищем коммит (Р7)

**На вынос:** слайд-шпаргалка + ссылка на материалы (статья на Хабре + скрипты)

### Git bisect заслуживает 60 секунд, а не 15

Текущее: "натравили bisect, справился, привёл к коммиту" — это 15 секунд.

Нужно: показать ПРИНЦИП git bisect для аудитории, которая его не знала.

Суть git bisect в 60 секунд:
1. Даёшь ему "плохой" коммит (баг есть) и "хороший" (бага нет)
2. Он выбирает коммит посередине, ты тестируешь — баг есть или нет?
3. Он отсекает половину и повторяет
4. За log2(N) шагов находит конкретный коммит из N
5. Для 50 000 коммитов — 16 проверок. При 30-секундном воспроизведении — 8 минут.

Это двоичный поиск по истории коммитов. Тестировщики Heisenbug знают двоичный поиск, но многие не знают, что git умеет делать это автоматически.

## Предложение: расширить git bisect в Р7 + переработать Р8

### Р7: расширить git bisect до ~60 сек
Добавить: принцип двоичного поиска, "50 000 коммитов → 16 проверок", связь с 30-секундным воспроизведением.

### Р8: вместо абстрактного "алгоритма" — конкретный "набор инструментов"
Переформулировать 6 шагов как КОНКРЕТНЫЕ инструменты с однострочным описанием:
1. **dmesg** — смотри не только логи приложения
2. **Паттерн** — что общего у всех падений
3. **Гипотеза → тест** — метод исключения
4. **Минимальный сценарий** — сократи до 30 секунд
5. **git bisect** — двоичный поиск по истории коммитов
6. **"Стеночки"** — иди глубже, любой чёрный ящик можно сделать белым
