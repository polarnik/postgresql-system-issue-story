# Три условия одновременно. По отдельности — безобидны.

# Условие 1: XFS — стандарт для больших баз

- Блоки до 64 КБ, параллельная работа с журналами
- В продакшне с PostgreSQL — **очень часто**
- На ext4 баг **не воспроизводится**

<!--
Проблемный файл должен лежать на XFS. Если перенести его на ext4 — баг не воспроизводится. XFS — это файловая система, которую часто выбирают для больших баз данных. Она хорошо работает с большими файлами, поддерживает блоки до 64 килобайт, умеет параллельно работать с журналами. Поэтому в продакшне с PostgreSQL она встречается очень часто. Если у вас ext4 — этот конкретный баг вас не касается. Но не потому что ext4 лучше, а просто потому что механизм другой.
-->

---

# Условие 2: init_on_free — обнуление памяти

<!--
Это параметр ядра Linux. Что он делает? Он обнуляет содержимое страниц памяти при их освобождении. То есть, когда процесс отдаёт память обратно системе, ОС заполняет эту память нулями. Зачем? Чтобы следующий процесс не мог подглядеть чужие данные.
-->

---

# Debian: выключен. Российские ОС: включён. ФСТЭК.

<!--
И вот что интересно: в Debian этот параметр по умолчанию выключен. А в российских дистрибутивах — включён. Потому что для сертификации ФСТЭК нужно обнулять память при освобождении. То есть в Debian баг не воспроизводится, а в российских дистрибутивах — воспроизводится. Подсказка на букву «Б» — безопасность.
-->

# Защита данных уничтожает код

```
mem auto-init: heap free:on
```

<!--
Ирония в том, что параметр, созданный для безопасности — «обнулять память» — приводит к тому, что обнуляется код. Вот те самые нули вместо инструкций — это оно.
Если хотите проверить, включён ли он у вас — ищите в логах mem auto-init: heap free:on. Вот если on — значит, включён.
-->

# Условие 3: давление на память + mmap

<!--
Когда памяти мало, ядро начинает освобождать страницы — это называется memory reclaim. Ну, грубо говоря, ОС ищет, что можно выгрузить из памяти, чтобы освободить место. И среди прочего она может выгрузить страницы, в которые были загружены файлы с диска.
А вот дальше — самое коварное. Когда вы запускаете любую программу — хоть Postgres, хоть ls — ОС не копирует весь файл программы в память. Она отображает его. Это называется mmap. То есть ОС говорит: «вот этот файл на диске — считай, что он уже в памяти, я подгружу нужные куски по мере обращения».
-->

# Ядро — не список, а граф

![bg w:90%](img/git-kernel-graph.svg)

<!--
Почему? Потому что ядро Linux — это не линейная цепочка коммитов. Это граф. Очень развесистый и ветвистый граф. Там есть ветки, которые были начаты за месяцы до релиза и замержены в самый последний момент.
Если наш баг влился через ветку подсистемы памяти — через коммит b2, скажем — то обычный git log может выдать вам коммит a2 из подсистемы сети как «средний». Вы его проверите — бага нет. И пойдёте по ложному следу. Потому что a2 просто не содержит кода из ветки памяти — он из другой подсистемы, сделан в то же время, но не имеет отношения к нашей проблеме.
Студент попал именно в эту ловушку. Он думал, что история — это линейная цепочка. А она — граф с тысячами переплетений.

# **git bisect**: поиск бага по графу за минуты

```bash
git bisect start
git bisect good v5.17     # бага нет
git bisect bad v5.18      # баг есть
git bisect run ./test.sh  # → 56a4d67c264e
```

<!--
И вот тут на сцену выходит git bisect.
Этот инструмент создал Линус Торвальдс, когда в 2005 году писал Git. Он понимал: при скорости разработки ядра — сотни патчей в день — регрессии неизбежны. И нужен инструмент, который превращает поиск бага из искусства в математическую процедуру.
Bisect работает не со списком, а с графом. Он понимает связи между ветками и мерж-коммитами. На каждом шаге он выбирает коммит, который делит граф достижимых коммитов на максимально равные части. Это не просто «возьми середину списка» — это топологический анализ.
Скрипт test.sh — наш тест воспроизведения. Возвращает 0 — коммит чистый. Возвращает 1 — баг есть. Возвращает 125 — коммит не собирается, пропустить.
-->

# Проверьте свои серверы

- Ядро **5.18 — 6.8** + **XFS** + **init_on_free** = обратитесь к вендору ОС
- Фикс в **6.9**, но **не помечен** как исправление
- **Бэкпорта нет**: на LTS-ядрах фикс сам не придёт

<!--
Второе — что делать прямо сейчас. Если у вас Linux-серверы — проверьте три вещи. Ядро от 5.18 до 6.8? Файловая система XFS? Параметр init_on_free включён? Если три из трёх — обратитесь к поставщику вашей ОС.
И вот тут есть неприятный нюанс. Фикс живёт в ядре 6.9, но он не помечен как исправление этого бага. Никто в upstream не связал патч с проблемой. А значит — бэкпорта в старые ядра нет. Если вы сидите на LTS-ядре вроде 5.15 или 6.1 — фикс к вам сам не придёт. Ваш вендор может просто не знать, что это нужно перенести. Вот это ловушка: баг уже исправлен в будущем, но в вашем настоящем он продолжает работать.
Что можно сделать? Обратиться к вендору ОС — возможно, конкретно в вашем ядре это уже починили. Если нет — как вариант, мигрировать на ext4 или указать init_on_free=0 в аргументах загрузки. Но на свой страх и риск — не забывайте, что init_on_free включают по требованиям безопасности. Тут нужен баланс.
И не делайте резких, необдуманных действий. Главное — не бежать обновлять прод сгоряча.
-->
