# Deep Dive: Git Bisect как инструмент расследования

**Секция доклада / отдельный материал**
**Контекст:** Мы использовали git bisect чтобы найти коммит в ядре Linux (тысячи коммитов между релизами), который вносил баг. Воспроизведение занимало 30 секунд. Bisect нашёл коммит за ~15 проверок.

**Источник:** Официальная документация git — https://git-scm.com/docs/git-bisect

---

## 1. Что такое git bisect

### Суть в одном предложении
Git bisect — это двоичный поиск по истории коммитов. Ты указываешь «здесь баг есть» и «здесь бага нет», а git находит конкретный коммит, который внёс проблему.

### Как это работает (схема)

```
История коммитов (упрощённо, 10 штук):

A --- B --- C --- D --- E --- F --- G --- H --- I --- J
✓                                                     ✗
good                                                  bad

Шаг 1: git bisect start
        git bisect good A
        git bisect bad J
        → git переключает на E (середина)

Шаг 2: тестируем E → баг есть
        git bisect bad
        → git переключает на C (середина A...E)

Шаг 3: тестируем C → бага нет
        git bisect good
        → git переключает на D (середина C...E)

Шаг 4: тестируем D → баг есть!
        git bisect bad
        → git сообщает: "D is the first bad commit"

Итого: 4 шага вместо просмотра 10 коммитов.
```

### Математика

Количество шагов — это `ceil(log₂(N))`, где N — число коммитов в диапазоне:

| Коммитов | Шагов | При 30 сек/шаг | При 2 мин/шаг |
|----------|-------|----------------|---------------|
| 100 | 7 | 3.5 мин | 14 мин |
| 1 000 | 10 | 5 мин | 20 мин |
| 10 000 | 14 | 7 мин | 28 мин |
| 15 000 | 14 | 7 мин | 28 мин |
| 50 000 | 16 | 8 мин | 32 мин |

Типичный релизный цикл ядра Linux — 13 000–16 000 коммитов. Это ~14 шагов bisect. При 30-секундном воспроизведении — 7 минут.

## 2. Автоматический режим: git bisect run

Ручной bisect требует человека на каждом шаге. Но есть автоматический режим:

```bash
git bisect start
git bisect bad v5.18
git bisect good v5.17
git bisect run ./test-script.sh
```

### Коды возврата (из официальной документации)

Скрипт для `git bisect run` должен возвращать:

| Код | Значение | Что делает bisect |
|-----|----------|-------------------|
| **0** | good — бага нет | Отмечает коммит как good, продолжает поиск |
| **1–124** | bad — баг воспроизвёлся | Отмечает коммит как bad, продолжает поиск |
| **125** | skip — невозможно протестировать | Пропускает коммит, выбирает соседний |
| **126** | (зарезервирован POSIX: command found but not executable) | Трактуется как bad |
| **127** | (зарезервирован POSIX: command not found) | Трактуется как bad |
| **128–255** | **аварийное завершение** | **Прерывает весь процесс bisect** |

Из документации git:
> *"Any other exit code will abort the bisect process. It should be noted that a program that terminates via exit(-1) leaves $? = 255, as the value is chopped with & 0377."*

**Важный подводный камень:** Если ваш тестовый скрипт падает с сигналом (например, segfault самого скрипта), shell возвращает 128 + номер сигнала (SIGSEGV=11 → код 139). Это **прервёт bisect**, а не отметит коммит как bad. Нужно ловить такие ситуации:

```bash
#!/bin/bash
./reproduce-test.sh
STATUS=$?

# Защита от сигналов: 128+ → трактуем как bad, не даём прервать bisect
if [ $STATUS -ge 128 ]; then
    echo "Test crashed with signal $(($STATUS - 128)), treating as bad"
    exit 1
fi
exit $STATUS
```

### Пример тестового скрипта для ядра

```bash
#!/bin/bash
set -e

# Шаг 1: собираем ядро
make -j$(nproc) 2>/dev/null
if [ $? -ne 0 ]; then
    echo "Build failed — skipping"
    exit 125
fi

# Шаг 2: устанавливаем в виртуалку и перезагружаемся
# (здесь специфичная для стенда логика: qemu, kexec, или модули)

# Шаг 3: запускаем тест воспроизведения
timeout 60 ./reproduce-xfs-bug.sh
RESULT=$?

if [ $RESULT -ge 128 ]; then
    exit 1   # тест упал по сигналу → bad
elif [ $RESULT -ne 0 ]; then
    exit 1   # тест провалился → bad
else
    exit 0   # тест прошёл → good
fi
```

## 3. Подводные камни

### Камень 1: Коммит не собирается

**Проблема:** В больших проектах промежуточные коммиты могут не компилироваться. В ядре Linux это особенно актуально — мерж-коммиты могут объединять код, который по отдельности не собирается.

**Решение:** Код возврата 125 (skip).

```bash
make -j$(nproc) 2>/dev/null || exit 125
```

**Что делает bisect при skip:** Выбирает ближайший соседний коммит. Из документации:
> *"However, if you skip a commit adjacent to the one you are looking for, Git will be unable to tell exactly which of those commits was the first bad one."*

**Последствие:** При большом количестве skip подряд bisect может не определить точный коммит-виновник. Он скажет что-то вроде: *"There are only skipped commits left to test. The first bad commit could be any of: abc1234, def5678, ghi9012"*. В этом случае нужно проверить указанные коммиты вручную.

**Worst case:** Если все коммиты в диапазоне кроме good и bad дают skip — bisect деградирует до линейного перебора оставшихся кандидатов.

### Камень 2: Нелинейная история с мержами

**Проблема (из рассказа Михаила):** "Линуксовое ядро очень страшным образом мержится. Это нелинейная штука, развесистая и ветвистая."

```
      feature-A ─── a1 ─── a2 ─── a3 ──┐
     /                                   \
main ─── M1 ───────────────────────────── M2 ─── M3
     \                                   /
      feature-B ─── b1 ─── b2 ──────────┘
```

Bisect может переключиться на коммит a2 из ветки feature-A, который не содержит кода из feature-B. Тест может дать неожиданный результат — не потому что баг ещё не внесён, а потому что контекст другой (не хватает кода из другой ветки).

**Решение — опция `--first-parent`:**

```bash
git bisect start --first-parent
git bisect bad v5.18
git bisect good v5.17
```

Из документации git:
> *"Follow only the first parent commit upon seeing a merge commit. In detecting regressions introduced through the merging of a branch, the merge commit will be identified as introduction of the bug and its ancestors will be ignored."*
> *"This option is particularly useful in avoiding false positives when a merged branch contained broken or non-buildable commits, but the merge itself was OK."*

С `--first-parent` bisect тестирует только основную линию коммитов (мерж-коммиты). Если виновник — мерж-коммит M2, bisect его найдёт. Потом можно вручную бисектить содержимое мержа.

**Когда `--first-parent` не помогает:** Если баг внесён не мержем, а конкретным коммитом внутри ветки — `--first-parent` покажет на мерж, а не на конкретный коммит. Тогда нужен второй раунд bisect внутри ветки.

### Камень 3: Плавающий баг — не всегда воспроизводится

**Проблема:** Если баг плавающий (как наш — зависит от memory pressure, timing), один запуск теста может дать ложный результат. False negative (баг есть, но тест не поймал) — самый опасный: bisect уйдёт в неправильную половину.

**Решение:** Запускать тест N раз. Считать good только если ВСЕ N проходят чисто:

```bash
#!/bin/bash
RUNS=5
for i in $(seq 1 $RUNS); do
    timeout 60 ./reproduce-test.sh
    RESULT=$?
    if [ $RESULT -ge 128 ]; then
        exit 1  # crash → bad
    elif [ $RESULT -ne 0 ]; then
        exit 1  # fail → bad
    fi
done
exit 0  # все N прошли → good
```

**Компромисс:**

| Запусков | Время на шаг | 14 шагов bisect | P(false negative) при 90% воспр. |
|----------|-------------|-----------------|----------------------------------|
| 1 | 30 сек | 7 мин | 10% на шаг |
| 3 | 1.5 мин | 21 мин | 0.1% на шаг |
| 5 | 2.5 мин | 35 мин | 0.001% на шаг |
| 10 | 5 мин | 70 мин | 0.000000001% |

**Наш подход:** Воспроизведение >90% стабильности за 30 секунд. Одного запуска хватало для большинства шагов. На подозрительных коммитах (рядом с границей good/bad) делали 3-5 запусков для перепроверки.

**Математика:** При 90% воспроизводимости и 14 шагах bisect, вероятность хотя бы одного false negative: 1 − 0.9¹⁴ ≈ 77%. То есть без повторных запусков ошибка почти гарантирована! При 3 запусках на шаг: 1 − 0.999¹⁴ ≈ 1.4%. При 5: практически 0%.

### Камень 4: Тест проверяет не то, что нужно

**Проблема:** Тест может давать false positive (бага нет, но тест падает по другой причине) и false negative (баг есть, но тест не ловит).

Примеры:
- Тест проверяет exit code приложения, а баг не всегда приводит к ненулевому exit code (как наше «воскрешение» — процесс продолжает работу)
- Тест зависит от timing — на быстрых коммитах работает, на медленных — нет
- Тест ловит другой баг, который тоже есть в этом диапазоне коммитов

**Решение:**
1. Тест должен проверять **конкретный симптом** (у нас — нули вместо единиц в mmap-файле), а не побочные эффекты
2. После bisect — **вручную проверить** найденный коммит и ±3 соседних
3. **Прочитать diff** найденного коммита — имеет ли он смысл как root cause?
4. Если diff не объяснимый — скорее всего тест ненадёжен. Нужно пересмотреть критерий

### Камень 5: Среда отличается между коммитами

**Проблема:** Баг может зависеть от конфигурации сборки, параметров ядра, версии компилятора. Если среда меняется между коммитами — результаты ненадёжны.

Конкретно для ядра Linux:
- `.config` может конфликтовать с новыми опциями (появляется `CONFIG_NEW_FEATURE`, которой не было в старом коммите)
- Новые версии GCC могут оптимизировать код иначе
- Модули ядра могут не загрузиться на другом ядре

**Решение:**
- Фиксировать `.config` — использовать `make olddefconfig` для адаптации конфига к каждому коммиту
- Фиксировать версию GCC (тот же компилятор для всех коммитов)
- Виртуалка со snapshot — откатываться к чистому состоянию перед каждым тестом
- Для ядра: использовать `kexec` или VM reboot для быстрой загрузки нового ядра

**Время сборки ядра (ориентир):**
- Полная сборка без кэша: 15–45 мин (зависит от конфигурации и CPU)
- Инкрементальная (`make -j$(nproc)` после small diff): 1–5 мин
- С `ccache`: первая сборка обычная, последующие — 2–5 мин
- Minimal config (`make tinyconfig`): 2–5 мин полная

## 4. Git bisect для тестирования производительности

### Можно ли так искать регрессии производительности?

Да. Из официальной документации git:
> *"For example, if you are looking for a commit that introduced a performance regression, you might use `git bisect start --term-old fast --term-new slow`"*

Git сам поддерживает пользовательские термины для этого сценария.

### Пользовательские термины

```bash
# Вместо good/bad — fast/slow
git bisect start --term-old fast --term-new slow
git bisect fast v5.17    # эта версия была быстрой
git bisect slow v5.18    # эта — медленная
git bisect run ./perf-test.sh
```

Или альтернативный синтаксис с old/new:
```bash
git bisect start
git bisect old v5.17     # до регрессии
git bisect new v5.18     # после регрессии
```

### Скрипт для тестирования производительности

```bash
#!/bin/bash
THRESHOLD=10000   # ops/sec — порог регрессии
RUNS=5

# Собираем
make -j$(nproc) 2>/dev/null || exit 125

# Несколько запусков, берём медиану
RESULTS=()
for i in $(seq 1 $RUNS); do
    R=$(./benchmark.sh | grep "ops/sec" | awk '{print $1}')
    RESULTS+=($R)
done

# Медиана (средний элемент отсортированного массива)
MEDIAN=$(printf '%s\n' "${RESULTS[@]}" | sort -n | sed -n '3p')

echo "Median: $MEDIAN ops/sec (threshold: $THRESHOLD)"

if [ "$MEDIAN" -lt "$THRESHOLD" ]; then
    exit 1   # slow
else
    exit 0   # fast
fi
```

### Подводные камни для производительности

#### Камень A: Шум измерений

**Проблема:** Производительность — шумная метрика. Один и тот же коммит может показать 10 500 ops/sec и 9 800 ops/sec на разных запусках. Если порог 10 000 — результат случаен.

**Решение:**
1. **Медиана, не среднее** — медиана устойчива к выбросам
2. **Порог с запасом** — если ожидаемая регрессия 20%, ставить порог на 15%
3. **Изоляция среды:** отключить turbo boost (`echo 1 > /sys/devices/system/cpu/intel_pstate/no_turbo`), привязать к ядрам (`taskset -c 0-3`), отключить hyperthreading, минимизировать фоновые процессы
4. **CPU-bound бенчмарки стабильнее IO-bound** — меньше зависимость от кэша и scheduling

#### Камень B: Регрессия меньше шума

**Проблема:** Если регрессия 5%, а шум ±3% — bisect будет ошибаться на каждом втором шаге.

**Решение:**
1. Увеличить количество запусков (10-20) для снижения дисперсии медианы
2. Использовать статистический тест (например, Mann-Whitney U) вместо простого порога
3. Или: отказаться от bisect для мелких регрессий — использовать линейное профилирование двух версий и сравнение flame graphs

#### Камень C: Постепенная деградация

**Проблема:** Bisect ищет ОДИН коммит — точку перехода. Но если производительность деградировала постепенно, серией коммитов по 1% каждый:

```
A(100%) → B(99%) → C(98%) → D(97%) → ... → J(85%)
```

Ни один коммит не даёт резкого перехода. Bisect не найдёт виновника.

**Решение — двухфазный подход:**

```
Фаза 1: Линейный прогон (каждый 100-й коммит):

ops/sec
10000 ●─●─●─●─●─●─┐
                    ●─●─●─┐
                           ●─●─●─● 8500
                    ↑
                    Основная деградация здесь
                    
Фаза 2: Bisect внутри найденного участка (~100 коммитов)
```

1. Сначала линейный прогон по N равномерно распределённым точкам
2. Построить график: ось X — коммиты, ось Y — производительность
3. Найти участки максимального падения
4. Запустить bisect внутри каждого участка

#### Камень D: Результат зависит от данных

**Проблема:** В PostgreSQL производительность запросов зависит от данных, индексов, планов выполнения. Новый коммит может изменить plan выбора, и это выглядит как регрессия на одних данных и как улучшение на других.

**Решение:**
1. Фиксировать тестовые данные (один и тот же дамп или seed для генератора)
2. Тестировать **несколько паттернов нагрузки** (OLTP, batch, mixed)
3. В скрипте бисекции — считать bad, только если **все паттерны** просели
4. Перед bisect: убедиться что регрессия воспроизводится стабильно на обеих граничных версиях

#### Камень E: Время сборки

**Проблема:** Если проект собирается 30 минут, то 14 шагов bisect = 7 часов только на сборку, плюс тестирование.

**Решение по убыванию затрат:**
1. **ccache** — кэширование результатов компиляции. Первая сборка обычная, последующие 2–5× быстрее
2. **Инкрементальная сборка** — `make -j$(nproc)` пересобирает только изменившиеся файлы
3. **Минимальная конфигурация** — для ядра: собирать только нужные модули, отключить всё лишнее
4. **Готовые артефакты из CI** — если в CI собирается каждый коммит, можно скачивать бинарники вместо сборки. Но для ядра Linux такого CI обычно нет
5. **Распределённая бисекция** — собирать и тестировать на нескольких машинах параллельно (нестандартно, но возможно)

## 5. Неочевидные применения git bisect

### 5.1 Поиск коммита, который ПОЧИНИЛ баг

Стандартный bisect ищет "первый плохой коммит". Но можно искать и "первый хороший" — коммит, который починил баг.

Два способа:

**Способ 1: пользовательские термины**
```bash
git bisect start --term-old broken --term-new fixed
git bisect broken v5.18   # здесь баг есть
git bisect fixed v6.9     # здесь починен
git bisect run ./test-for-fix.sh
# test-for-fix.sh: exit 0 = баг починен (fixed), exit 1 = баг ещё есть (broken)
```

**Способ 2: old/new**
```bash
git bisect start
git bisect old v5.18      # старое состояние (баг есть)
git bisect new v6.9       # новое состояние (починен)
```

Мы использовали этот подход, чтобы найти коммит-фикс `bc2ff4c` в ядре 6.9.

### 5.2 Bisect-мышление за пределами git

Принцип двоичного поиска можно применять к любому пространству с бинарным критерием:

**Bisect по конфигурации:**
Если баг зависит от параметров (как наш init_on_free), можно "бисектить" пространство параметров:
1. Выписать все подозрительные параметры (10 штук)
2. Включить половину, выключить половину. Баг есть?
3. Сузить до 5 параметров, потом до 2, потом до 1

**Bisect по версиям зависимостей:**
```bash
# В какой версии библиотеки появилась проблема?
# Линейно (для малого числа версий):
for version in 1.0 1.1 1.2 1.3 1.4 1.5; do
    pip install library==$version
    ./test.sh && echo "$version: OK" || echo "$version: BROKEN"
done

# Или двоичным поиском для большого числа версий
```

**Bisect по условиям воспроизведения:**
- При каком минимальном объёме памяти баг начинает проявляться? (4GB → 2GB → 1GB → ...)
- При каком числе параллельных потоков? (1 → 2 → 4 → 8 → ...)
- При каком размере данных? (1GB → 10GB → 100GB → ...)

### 5.3 Комбинация bisect с другими инструментами

**bisect + perf:**
На каждом шаге bisect, кроме проверки good/bad, можно собирать perf-данные. После bisect — сравнить flame graphs соседних коммитов (последний good vs первый bad).

**bisect + strace:**
Записать syscall trace на good и bad коммитах, сравнить — какие вызовы появились/исчезли.

**bisect + git blame:**
После нахождения проблемного коммита — `git blame` на изменённых строках, чтобы понять контекст и историю этого кода.

## 6. Резюме: когда использовать git bisect

### Подходит

- Есть чёткий критерий good/bad (или порог производительности)
- Воспроизведение автоматизировано и занимает < 5 минут (идеально < 1 минуты)
- Диапазон поиска > 50 коммитов (при меньшем — быстрее руками)
- Есть автоматический тест с надёжным результатом

### Не подходит

- Постепенная деградация без чёткого перехода (нужен двухфазный подход)
- Воспроизведение нестабильно (< 50% успеха) и нет возможности улучшить
- Сборка проекта занимает часы и нет кэширования
- Баг зависит от внешних условий, которые не фиксируются (время суток, сетевая нагрузка)

### Чек-лист перед запуском

```
[ ] known good коммит проверен — тест возвращает 0
[ ] known bad коммит проверен — тест возвращает 1
[ ] тест корректно обрабатывает сигналы (не возвращает 128+)
[ ] тест обрабатывает ошибки сборки (возвращает 125)
[ ] среда фиксирована: компилятор, конфигурация, данные, VM snapshot
[ ] для плавающих багов: несколько запусков на шаг (минимум 3)
[ ] для производительности: медиана из 5+ запусков, порог с запасом
[ ] время одного шага (сборка + тест) известно и приемлемо
```
