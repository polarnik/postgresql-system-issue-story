# Deep Dive: Git Bisect как инструмент расследования

**Секция доклада / отдельный материал**
**Контекст:** Мы использовали git bisect чтобы найти коммит в ядре Linux (50 000+ коммитов), который вносил баг. Воспроизведение занимало 30 секунд. Bisect нашёл коммит за 16 проверок (~8 минут).

---

## 1. Что такое git bisect

### Суть в одном предложении
Git bisect — это двоичный поиск по истории коммитов. Ты указываешь «здесь баг есть» и «здесь бага нет», а git находит конкретный коммит, который внёс проблему.

### Как это работает (схема)

```
История коммитов (упрощённо):

A --- B --- C --- D --- E --- F --- G --- H --- I --- J
✓                                                     ✗
good                                                  bad

Шаг 1: git bisect start
        git bisect bad J    (баг есть)
        git bisect good A   (бага нет)
        → git переключает на E (середина)

Шаг 2: тестируем E → баг есть
        git bisect bad
        → git переключает на C (середина A...E)

Шаг 3: тестируем C → бага нет
        git bisect good
        → git переключает на D (середина C...E)

Шаг 4: тестируем D → баг есть!
        git bisect bad
        → git говорит: "D is the first bad commit"

Итого: 4 шага вместо 10 ручных проверок.
```

### Математика

| Коммитов | Проверок (log₂) | При 30 сек/проверку |
|----------|-----------------|---------------------|
| 100 | 7 | 3.5 мин |
| 1 000 | 10 | 5 мин |
| 10 000 | 14 | 7 мин |
| 50 000 | 16 | 8 мин |
| 100 000 | 17 | 8.5 мин |

В нашем случае: ~50 000 коммитов между ядрами 5.17 (good) и 5.18 (bad) → 16 проверок → 8 минут.

## 2. Автоматический режим: git bisect run

Ручной bisect требует человека на каждом шаге. Но есть автоматический режим:

```bash
git bisect start
git bisect bad v6.8
git bisect good v5.17
git bisect run ./test-script.sh
```

`test-script.sh` должен вернуть:
- **0** — баг НЕ воспроизводится (good)
- **1-124, 126-127** — баг воспроизводится (bad)
- **125** — невозможно протестировать этот коммит (skip)

### Наш тестовый скрипт (упрощённо)

```bash
#!/bin/bash
# Собираем ядро с этим коммитом
make -j$(nproc) 2>/dev/null
if [ $? -ne 0 ]; then
    exit 125  # не собралось — skip
fi

# Устанавливаем и перезагружаемся в новое ядро
# (здесь магия с виртуалкой)

# Запускаем тест воспроизведения
timeout 60 ./reproduce-test.sh
if [ $? -eq 0 ]; then
    exit 0   # бага нет — good
else
    exit 1   # баг есть — bad
fi
```

## 3. Подводные камни

### Камень 1: "Ядро не собирается на этом коммите"

**Проблема:** Промежуточные коммиты в больших проектах (особенно в ядре Linux) могут не собираться. Мерж-коммиты могут ломать сборку.

**Решение:** Код возврата **125** (skip). Git bisect пропустит этот коммит и выберет соседний.

```bash
# Если сборка упала — skip
make -j$(nproc) 2>/dev/null || exit 125
```

**Последствие:** При большом количестве skip'ов bisect теряет эффективность — вместо log₂(N) может понадобиться больше шагов.

### Камень 2: "Нелинейная история с мержами"

**Проблема (из рассказа Михаила):** "Линуксовое ядро очень страшным образом мержится. Это нелинейная штука, развесистая и ветвистая."

**Что это значит:**
```
      feature-A ─── a1 ─── a2 ─── a3 ──┐
     /                                   \
main ─── M1 ───────────────────────────── M2 ─── M3
     \                                   /
      feature-B ─── b1 ─── b2 ──────────┘
```

Bisect может переключиться на коммит a2 из feature-A, который вообще не содержит кода из feature-B. Тест может вести себя иначе, не потому что баг ещё не внесён, а потому что контекст другой.

**Решение:**
- Использовать `git bisect skip` для коммитов, которые дают неоднозначные результаты
- Если bisect показывает мерж-коммит как виновника → вручную проверить обе ветки, которые мержились
- Опция `--first-parent` для тестирования только основной ветки:
  ```bash
  git log --first-parent --oneline v5.17..v5.18 | wc -l
  ```

### Камень 3: "Плавающий баг — не всегда воспроизводится"

**Проблема:** Если баг плавающий (как наш — зависит от memory pressure, timing), один запуск теста может дать ложный результат.

**Решение:** Запускать тест N раз и считать good только если ВСЕ N проходят чисто.

```bash
#!/bin/bash
RUNS=5
for i in $(seq 1 $RUNS); do
    timeout 60 ./reproduce-test.sh
    if [ $? -ne 0 ]; then
        exit 1  # bad: баг воспроизвёлся хотя бы раз
    fi
done
exit 0  # good: за N запусков баг не появился
```

**Компромисс:** Больше запусков → надёжнее, но дольше. При 5 запусках по 30 секунд = 2.5 мин на проверку. 16 шагов × 2.5 мин = 40 минут вместо 8. Всё равно быстрее, чем руками.

**Наш подход:** Мы добились воспроизведения за 30 секунд с высокой стабильностью (>90%). Поэтому одного запуска было достаточно в большинстве случаев, с перепроверкой на подозрительных коммитах.

### Камень 4: "Тест работает, но результат ненадёжен"

**Проблема:** В тесте могут быть false positive и false negative:
- **False negative** (баг есть, но тест не поймал): самый опасный. Bisect решит, что коммит good, и уйдёт в неправильную половину.
- **False positive** (бага нет, но тест говорит что есть): менее опасный, но тоже приводит к ложному виновнику.

**Решение:**
1. Тест должен проверять КОНКРЕТНЫЙ симптом (у нас — нули вместо единиц в mmap-файле), а не побочные эффекты
2. После bisect — вручную проверить найденный коммит и ±2 соседних
3. Прочитать diff найденного коммита — имеет ли он смысл как root cause?

### Камень 5: "Нужна та же среда для каждого коммита"

**Проблема:** Баг может зависеть от конфигурации сборки, параметров ядра, версии компилятора. Если среда отличается — результаты бисекции будут ненадёжными.

**Решение:**
- Фиксировать конфигурацию сборки (одинаковый .config для всех коммитов)
- Фиксировать версию компилятора (тот же gcc)
- Использовать виртуалку с snapshot — откатываться к чистому состоянию перед каждой проверкой
- Автоматизировать: скрипт развёртывания должен быть идемпотентным

## 4. Git bisect для тестирования производительности

### Можно ли так искать регрессии производительности?

**Да!** И это один из самых недооценённых вариантов использования. Вместо "баг есть / бага нет" критерий становится "производительность в норме / просела".

### Как это выглядит

```bash
#!/bin/bash
# Собираем, деплоим
make && ./deploy.sh

# Запускаем бенчмарк
RESULT=$(./benchmark.sh | grep "ops/sec" | awk '{print $1}')

# Порог: если ops/sec < 10000 — регрессия
THRESHOLD=10000
if [ "$RESULT" -lt "$THRESHOLD" ]; then
    exit 1  # bad: регрессия
else
    exit 0  # good: производительность в норме
fi
```

### Подводные камни для производительности

#### Камень A: "Шум измерений"

**Проблема:** Производительность — шумная метрика. Один и тот же коммит может показать 10500 ops/sec и 9800 ops/sec на разных запусках. Если порог 10000 — результат случаен.

**Решение:**
1. Запускать бенчмарк несколько раз, брать медиану
2. Порог выбирать с запасом (например, если ожидаем регрессию 20%, ставим порог на 15%)
3. Использовать статистический тест вместо простого порога

```bash
#!/bin/bash
# 5 запусков, берём медиану
RESULTS=()
for i in $(seq 1 5); do
    R=$(./benchmark.sh | grep "ops/sec" | awk '{print $1}')
    RESULTS+=($R)
done

# Сортируем и берём медиану (3-й элемент из 5)
MEDIAN=$(printf '%s\n' "${RESULTS[@]}" | sort -n | sed -n '3p')

if [ "$MEDIAN" -lt "$THRESHOLD" ]; then
    exit 1
else
    exit 0
fi
```

#### Камень B: "Регрессия маленькая, а бисекция грубая"

**Проблема:** Если регрессия 5% (было 10000 ops/sec, стало 9500), а шум измерений ±3% — bisect будет часто ошибаться на границе.

**Решение:**
1. Увеличить количество запусков (10-20 вместо 5)
2. Использовать более стабильный бенчмарк (CPU-bound лучше, чем IO-bound)
3. Изолировать среду: отключить turbo boost, привязать к ядрам (taskset), отключить фоновые процессы
4. Или: вместо bisect использовать полное профилирование двух версий (до/после) и сравнивать flame graphs

#### Камень C: "Постепенная деградация vs резкий скачок"

**Проблема:** Bisect ищет ОДИН коммит — точку перехода от good к bad. Но что если производительность деградировала постепенно, серией коммитов? Каждый по 1%, а вместе — 15%.

```
A(100%) → B(99%) → C(98%) → D(97%) → ... → J(85%)
```

Bisect не найдёт "виновника", потому что ни один коммит не даёт резкого перехода.

**Решение:**
1. Увеличить чувствительность порога (но тогда больше false positive из-за шума)
2. Использовать bisect итеративно: сначала найти диапазон из 100 коммитов, потом внутри него — из 10, потом из 3
3. Или: линейный прогон по N точкам (каждый 100-й коммит), построить график, найти участки деградации, потом bisect внутри участка

```
Линейный прогон (каждый 100-й коммит):

ops/sec
10000 ●─●─●─●─●─●─┐
                    ●─●─●─┐
                           ●─●─●─● 8500
                    ↑
                    Где-то здесь — основная деградация
                    → bisect этот участок
```

#### Камень D: "Результат зависит от данных"

**Проблема:** В PostgreSQL производительность запросов зависит от данных, индексов, planner decisions. Новый коммит может изменить план запроса, и это выглядит как регрессия, хотя на других данных — ускорение.

**Решение:**
1. Фиксировать тестовые данные (один и тот же дамп/seed)
2. Тестировать несколько паттернов нагрузки, а не один
3. В скрипте бисекции — проверять все паттерны, считать bad только если ВСЕ просели

#### Камень E: "Время сборки"

**Проблема:** Если проект собирается 30 минут (как ядро Linux с нуля), то 16 шагов bisect = 8 часов только на сборку.

**Решение:**
1. Инкрементальная сборка (make -j) — обычно 2-5 мин вместо 30
2. ccache — кэширование результатов компиляции
3. Для ядра: собирать только нужные модули
4. Для больших проектов: держать CI с артефактами для каждого коммита → вместо сборки скачиваем готовый билд

## 5. Неочевидные применения git bisect

### 5.1 Поиск коммита, который ПОЧИНИЛ баг
```bash
# "Перевёрнутый" bisect: good и bad меняются местами
git bisect start
git bisect good v6.9    # баг починен (= good для нас)
git bisect bad v5.18    # баг есть (= bad для нас)
```
Мы использовали это чтобы найти фикс — коммит, после которого баг перестал воспроизводиться.

### 5.2 Bisect по конфигурации, а не по коммитам
Идея: если баг зависит от параметров конфигурации (как init_on_free), можно "бисектить" пространство параметров. Не через git, а через аналогичный подход:
1. Выписать все подозрительные параметры
2. Включать/выключать половину за раз
3. Сужать до конкретного параметра

Это метод "bisect мышления", даже без git.

### 5.3 Bisect по версиям зависимостей
```bash
# Проверяем, в какой версии библиотеки появилась проблема
for version in 1.0 1.1 1.2 1.3 1.4 1.5; do
    pip install library==$version
    ./test.sh && echo "$version: GOOD" || echo "$version: BAD"
done
```
Можно автоматизировать двоичным поиском.

### 5.4 Bisect по времени (для воспроизведения плавающих багов)
Если баг появляется "иногда" — можно использовать bisect для поиска условий:
- Bisect по размеру памяти (4GB, 2GB, 1GB — при каком начинает падать?)
- Bisect по числу параллельных процессов
- Bisect по размеру данных

## 6. Резюме: когда использовать git bisect

### Подходит

- Есть чёткий критерий good/bad (или порог производительности)
- Воспроизведение < 5 минут (идеально < 1 минуты)
- Диапазон поиска > 50 коммитов
- Есть автоматический тест

### Не подходит

- Постепенная деградация без чёткого перехода
- Воспроизведение нестабильно (< 50% успеха)
- Сборка проекта занимает часы
- Баг зависит от внешних условий, которые не фиксируются

### Чек-лист перед запуском

- [ ] Тестовый скрипт возвращает 0 (good), 1 (bad), 125 (skip)
- [ ] Known good коммит проверен — тест действительно проходит
- [ ] Known bad коммит проверен — тест действительно падает
- [ ] Среда фиксирована (компилятор, конфигурация, данные)
- [ ] Для плавающих багов — несколько запусков на проверку
- [ ] Для производительности — медиана, а не среднее; порог с запасом

В контексте нашего расследования:

| Шаг | Что мы делали | Подводный камень |
|-----|--------------|-----------------|
| Воспроизведение | Сократили с раз в неделю до 30 сек | Камень 3 (плавающий баг) |

| Known bad | Ядро 5.18 — баг есть | — |

Без 30-секундного воспроизведения bisect был бы невозможен. Без bisect мы бы руками искали в 50 000 коммитов. Два инструмента усилили друг друга.
