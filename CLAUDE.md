## Контекст

Готовим доклад для конференции **Heisenbug** от имени **Михаила Жилина** — инженера по производительности с 15+ летним стажем (Postgres Professional). Доклад на **40 минут** рассказа.

**Ключевое сообщение:** Когда система падает "без причин" — причина есть всегда, и чтобы её найти, нужно идти вглубь стека: от приложения через ОС до ядра.

**Аудитория:** Heisenbug — практическая конференция для тестировщиков, QA-автоматизаторов, инженеров по производительности в России. Люди приходят за практикой. Много пользователей PostgresPro.

- `#0061e0` → `#cc7e19` (тёмно-оранжевый)
- `#00296f`, `#01629c` → `#333` (тёмно-серый)
- `stroke:#eeeeee` → `stroke:#333`

### Mermaid themeVariables (для timeline)
Для полного контроля цветов нужно задавать ВСЕ 12 переменных cScale0-11, cScaleLabel0-11, cScalePeer1-12 + fontFamily + lineColor. Если задать только cScale0 — Mermaid автогенерирует остальные с нежелательными цветами.

```
'fontFamily': 'verdana',
'lineColor': '#ff9e1f',
'cScale0': '#222',        // done: тёмный
'cScale4': '#ff9e1f',     // active: оранжевый
'cScaleLabel0': '#aaa',   // done: серый текст
'cScaleLabel4': '#000',   // active: чёрный текст на оранжевом
'cScaleLabel11': '#ff9e1f', // ВАЖНО: задаёт цвет стрелок!
'cScalePeer1': '#333',    // 1-indexed!
```
**Неочевидно:** `cScaleLabel11` управляет цветом стрелок на timeline, а не текстом 12-й секции. Эталонный файл — `timeline-5.mmd`.

## Правила работы с текстом

### Голос спикера
- Весь текст — **голосом Михаила Жилина**: разговорный, неформально-технический, с самоиронией
- Слова-связки: "вот", "ну", "то есть", "блин" (2-3 раза за доклад), "мы такие", "вот смотрите"
- Уменьшительные: "табличка", "базенка", "виртуалочка", "ссылочка", "салатики"
- **"Мы", а не "я"** — история команды, 5-6 человек
- Технические термины: **термин + "то есть" + простое объяснение**
- Историческое настоящее: "мы такие открываем dmesg, смотрим..."
- Литотес (преуменьшение): "немножко напряглись" = катастрофа

### Принципы подачи
- **Показывать, не поучать.** Михаил делится опытом ("вот как у нас было"), а не менторствует ("вам нужно делать так"). Слушатель сам делает выводы
- **Системы, не люди.** "Что-то не так настроено", а не "кто-то не так настроил". Проблемы — в компонентах и стечении обстоятельств
- **Не спойлерить.** Не раскрывать выводы до того, как слушатель прошёл путь вместе с рассказчиком

### Работа с источниками
- **Формулировки Михаила приоритетнее наших.** Если в `texts/init.md` или `texts/initial.meeting.summary.md` есть конкретные слова Михаила — использовать их, а не придумывать свои
- **Не выдумывать факты.** Если чего-то нет в источниках — не додумывать. Спросить или опустить
- **Не выдумывать сцены.** Конкретные моменты ("клиент присылает дамп", "кто-то из команды сказал") — только из источников
- **Проверять хронологию.** Порядок событий: воскрешение обнаружили рано (Р4), лечилку написали после понимания механизма (Р5). Команда сначала ОТПРАВИЛА "не наш баг", потом вернулась к проблеме.
- **Андрес Фройнд — из Microsoft**, не из PostgreSQL (контрибьютор PostgreSQL, но сотрудник Microsoft)
- **texts/fix.md содержит AI-формулировки**, не слова Михаила. Не копировать оттуда дословно.
- **Коммит-баг: +5/-5 строк в 3 файлах.** Не "шесть строчек" (неверно). Говорить "несколько строчек" или "крошечный diff".

## Чего НЕ делать (антипаттерны)

### В тексте доклада
- **НЕ академический стиль.** Никаких "таким образом", "следовательно", "дальнейшее расследование показало". Говорить: "то есть", "мы копали дальше и выяснилось"
- **НЕ мат.** Максимум "блин", "нафиг"
- **НЕ шутки на чей-то счёт.** Только самоирония
- **НЕ аналогии с профессиями.** Уборщицы, охранники — могут задеть людей этих профессий. Аналогии из мира вещей, механизмов, химии, природы
- **НЕ военная тематика.** Бронежилеты, гранаты, взрывчатка, окопы. В повестке дня и так много боевых действий
- **НЕ иерархии ролей.** Не противопоставлять "инженер vs оператор", "DBA vs разработчик"
- **НЕ put-down конкурентов.** Не упоминать конкретные продукты негативно
- **НЕ маркетинг.** Доклад — история инженерного расследования, не продвижение компании
- **НЕ выдумывать факты и сцены.** Не додумывать то, чего Михаил не говорил
- **НЕ менторствовать.** "Вот как у нас было", а не "вам нужно запомнить"
- **НЕ спойлерить.** Не говорить "самое полезное в докладе" до того, как это показано

### В диаграммах Mermaid
- **НЕ sequence diagram** для progressive reveal — нет индивидуальной стилизации
- **НЕ stateDiagram, ZenUML, architecture-beta** — нет classDef/linkStyle
- **НЕ менять текст между слайдами серии** — layout поплывёт, элементы сдвинутся
- **НЕ менять stroke-width между слайдами** — layout поплывёт
- **НЕ подписи на стрелках** — занимают место, нестабильны между файлами
- **НЕ длинные тексты в узлах** — 1-2 слова максимум
- **НЕ `<br/>` в узлах** — меняет высоту
- **НЕ flowchart LR с subgraph direction LR** — Mermaid игнорирует вложенный LR внутри LR
- **НЕ переменная толщина рамок** — всегда `stroke-width:2px`
- **НЕ забывать `style S1 fill:transparent,stroke:transparent`** — иначе видна рамка subgraph
- **НЕ задавать только cScale0** для timeline — Mermaid автогенерирует остальные 11 с нежелательными цветами. Задавать ВСЕ 12 cScale + cScaleLabel + cScalePeer.

## Правила для Mermaid-диаграмм

Полное руководство: **`docs/diagrams/README.md`**

### Типы диаграмм
- **flowchart** — для progressive reveal (диафильмов) и схем процессов. Единственный тип с classDef + linkStyle
- **gitGraph** — для визуализации истории коммитов, веток. Статичные слайды, стилизация через тему
- **timeline** — для хронологии расследования и навигации по докладу. Progressive reveal через постепенное добавление секций

### Паттерн progressive reveal (flowchart)
- **TB + subgraph LR** — два ряда под горизонтальный слайд (1280x720)
- Все файлы серии: **идентичный текст, идентичная структура**. Меняются только classDef назначения и linkStyle цвета
- **5 classDef**: done, active, pending, error, success (см. таблицу цветов выше)
- **stroke-width всегда 2px** — и в classDef, и в linkStyle
- **Прозрачные рамки subgraph**: `style S1 fill:transparent,stroke:transparent`
- **Проверка**: `grep -oE '\["[^"]+"\]' file-1.mmd` должен совпадать для всех файлов серии

### Паттерн timeline (5 этапов)
Все 6 файлов (timeline-1..5 + timeline-full) содержат **одинаковую структуру** с 5 этапами × 4 блока = 20 блоков. Различаются только цветами cScale.

| Этап | Разделы | Мин | Слайдов | Блоки |
|------|---------|-----|---------|-------|
| 30 декабря | R1+R2 | 4.3 | 11 | segfault, гипотеза, пинг-понг, копаем |
| Зацепки | R3+R4 | 5.9 | 15 | нули в dmesg, не Postgres, нагрузка, воскрешение |
| Три условия | R5 | 4.9 | 11 | XFS, init_on_free, pressure, лечилка |
| Доказательство | R6+R7 | 6.3 | 12 | 30 секунд, два теста, git bisect, коммит |
| Root cause | R8 | 5.4 | 9 | race condition, фикс 6.9, XZ Utils, бэкпорта нет |

Timeline появляется **ПОСЛЕ** того, как все блоки этапа рассказаны:
- timeline-1 → после R2
- timeline-2 → после R4
- timeline-3 → после R5
- timeline-4 → после R7
- timeline-5 → после R8
- timeline-full не используется в слайдах (используется timeline-5 как финальная)
